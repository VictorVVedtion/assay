# LLM 中转站独立验证中间层 — 设计文档 v0.1

> 一句话:一个**开源、自托管、信任最小化**的中间层,独立核验你从 LLM 中转站(API 转售/聚合)买到的**用量**和**模型身份**是否名副其实——不需要中转站配合,不需要交出 root 账号。
>
> 项目代号待定(候选见 §9)。本文档由深度研究(106 agent / 23 信源 / 25 条声明三票对抗验证)+ 架构推演合成。

---

## 1. 问题与威胁模型

中转站本质是**中间人(MITM)**,有两个它无法自证、买家无法核实的信任黑洞:

1. **用量真实性** — 按 token 计费,但计费数字是中转站自报。可虚报输入/输出 token,可对缓存命中照样计费。
2. **模型身份真实性** — 你点名旗舰模型(GPT-4o / Claude Opus / DeepSeek-R1),它可在后台静默降级成便宜模型、量化版,或套壳开源模型,照旗舰价计费。

**为什么 Cloudflare AI Gateway 这类方案解决不了**:它是托管观测台——要你交账号/root、闭源、不在数据流上做独立校验。它把"信任问题"从中转站转移到了 Cloudflare,而没有消除。

### 威胁模型(对手能力)

中转站是**潜在对手**。它**能**做:
- 虚报 `usage`(prompt/completion/total tokens)
- 缓存重放并计费
- 模型降级 / 量化 / 套壳
- 任意伪造 `model` 字段、`system_fingerprint`、`usageMetadata` 等一切元数据

它**不能**做(这是检测的根基):
- 伪造它**实际交付给你的输出 token 本身**——因为"省成本"要求它真的把便宜模型的 token 发给你,而那些 token 携带便宜模型的指纹。**省钱与完美伪装在经济上互斥**(想让输出分布像旗舰,就得真跑旗舰,等于没省)。

### 两类用户(同一二进制,不同挂载位置)

```
场景 A:买家审计中转站            场景 B:运营者审计自己的上游
你的 app                         你的 new-api(渠道 base_url 改指审计器)
   │                                │
   ▼  审计器在你这                   ▼  审计器在你这
[审计器] ───▶ 中转站的 new-api      [审计器] ───▶ 真·上游 / 更上游的中转站
```

原则:**审计器必须待在审计方自己掌控、与被审计方分离的信任域里。** 不能做成塞进中转站的模块(等于让狐狸自己装报警器)。

## 2. 核心设计原则(从两条"硬伤"推导出来)

两条硬伤:① 客户凭什么信我们的审计器;② 验证依赖的"正品基准"从哪来。二者本质是同一问题——**在谁都不可信的环境里,信任与地面真相从哪来**。答案定义了产品形态:

- **信任最小化,而非被信任** — 开源 + 自托管 + **可复现判定**。审计器只做"记录 + 确定性校验",任何人都能拿日志里的原始字节重跑同一套开源校验器得到同样结论。你信的不是我们的"结论",是"人人可复算"。
- **经济约束是基岩** — 见 §1 威胁模型。哪怕中转站把 logprob/metadata 全藏,它**必须交付的输出文本本身**就携带可分类指纹。
- **分层防御** — 免基准的便宜常态检测(floor)+ 依赖基准的深度审计(ceiling)。
- **被动全量优于主动探针** — 中转站最强的招是"探针识别"(只对体检请求发正品)。被动记录真实流量它防不住(无法区分"真实请求"和"我在记录真实请求")。
- **诚实标注不确定性** — 输出"与参考分布不一致 + 证据",**不轻易断言"诈骗"**(见 §8 归因鸿沟)。这既是法律自保,也是赢得信任的方式。

## 3. 研究结论:可用的检测方法

> 来源均为 2024–2025、对 2026 仍现行;但 tiktoken 映射、消息框架常量、各家分词器政策会漂移,实现须跟踪上游而非写死。

### 3A. 用量真实性

| 方法 | 适用 | 免基准 | 置信度 | 今天可实现 | 备注 |
|---|---|---|---|---|---|
| **tiktoken 本地重算 + Chat 框架数学** | OpenAI | ✅ 完全离线 | **高**(接近估计,非逐字节) | ✅ 立即 | 唯一可完全离线独立重算的厂商 |
| 缓存重放检测(微扰 prompt → 逐字节雷同) | 全部 | ✅ | **高(实锤)** | ✅ 立即 | |
| `count_tokens` 往返 | Claude | ❌ 往返 Anthropic(泄露 prompt、仅 input、仅估计) | 中 | ⚠️ 弱 | 无公开分词器 |
| 逆向分词器(javirandor/anthropic-tokenizer) | Claude | ✅ 离线但**保真度未验证** | 低–中 | ⚠️ 近似 | 仅作粗略代理 |
| `usageMetadata` 比对 / `countTokens` 往返 | Gemini | ❌ 转发数据可伪造 / 需往返 Google | 低–中 | ⚠️ 弱 | 无公开分词器 |

**关键:用量验证不对称性。** 只有 OpenAI 能完全离线独立核验(tiktoken 公开、MIT、`o200k_base`/`cl100k_base` 等按模型名确定性映射)。实现须复刻 Chat Completions 框架数学(`tokens_per_message=3`、`tokens_per_name=1`、回复 +3;旧版 `gpt-3.5-turbo-0301` 为 4/-1),且结果是"接近估计"非逐字节(function-calling 等会引入未计入 token)。**Claude/Gemini 无公开分词器是硬限制,不是实现缺口。** 跨厂商一律:**转发的 `usage`/metadata 视为不可信,能本地重算就本地重算。**

### 3B. 模型身份真实性

| 方法 | 论文 / 开源代码 | 免基准 | 置信度 | 今天可实现 | 备注 |
|---|---|---|---|---|---|
| **Model Equality Testing(MMD 文本分布检验,无需 logprobs)** | [arXiv 2410.20247](https://arxiv.org/abs/2410.20247) ICLR'25 / `i-gao/model-equality-testing` | ❌ 需正品参考分布(~200 样本) | **中–高**(77.4% 中位功效,量化好抓、微调难) | ✅ 现成代码,<$1/次 | **首选深度方法** |
| **LLMmap 主动指纹(8 次 query)** | [arXiv 2407.15847](https://arxiv.org/abs/2407.15847) AAAI'26 / `pasquini-dario/LLMmap` | ❌ 需训练候选集分类器 | 高(**闭集** >95%;开集低) | ✅ 现成代码 | 低预算常态探针 |
| 对抗感知审计套件(评测各法在攻击下表现) | [arXiv 2504.04715](https://arxiv.org/abs/2504.04715) UC Berkeley(Dawn Song)/ `sunblaze-ucb/llm-api-audit` | — | — | ✅ 参考实现 | 设计抗规避的圣经 |
| 分词器家族矛盾(流式 chunk 边界暴露分词器) | — | ✅(分词器–模型绑定是公开知识) | 中 | ✅ 立即 | 抓懒惰作弊;可被重分词规避 |
| 物理速度不可能 / 延迟·吞吐指纹 | [arXiv 2502.12150](https://arxiv.org/abs/2502.12150) 等 | ✅ priors | 低(辅助/粗筛) | ✅ 旁路 | |
| ~~logprob 分布指纹~~ | — | ❌ | **理论最强,实际被击败** | ❌ **放弃** | 见下方纠正 |
| ~~`system_fingerprint` / `model` 字段~~ | — | — | **无(可任意伪造)** | ❌ 不可信 | 见 §1 |

**⚠️ 一处对早期判断的诚实纠正:** 我先前说"logprob 指纹是最强单一信号"——**研究推翻了它的可用性**。logprob 方法理论最强,但在生产中被**GPU 浮点非结合性 + 批处理导致的推理非确定性**击败,且修复手段(batch-invariant kernel)是**服务端**的,外部审计器无法依赖(2504.04715 §4.2;Thinking Machines《Defeating Nondeterminism in LLM Inference》2025-09 印证)。**实际可用的深度方法是 MMD 文本分布检验(根本不需要 logprobs)。**

**另一处校准:** "软件方法根本不可靠"这一说法在验证中被**否决(0-3)**——文本样本法**确实有效**,只是查询密集。不要夸大不可靠性;但也别假设 LLMmap 对一切规避(prompt 包装、改温度)鲁棒——该鲁棒性声明同样被**否决(0-3)**。

## 4. 架构:三平面 + 证据日志

实时(用户在用)与深度验证(慢)能共存的**唯一办法是切开数据面与分析面**:

```
              用户 app ──实时请求──▶ ┌─────────────────────────┐
                                    │ 数据面 (透传,毫秒级)       │ ← 极简 / fail-open
                                    │ 转发 → 流式原样回传        │
                                    │ → 同时 tee 一份完整副本     │
                                    └─────────────────────────┘
                            转发 │                    │ 异步副本(不阻塞用户)
                                 ▼                    ▼
                            中转站→上游         ┌──────────────────────┐
                                              │ 分析面 (离线,可慢可深)   │
                                              │ token 重算/指纹/缓存检测  │
                                              └──────────┬───────────┘
                                                         ▼
                                              ┌──────────────────────┐
        探针面 (定时旁路) ──注入 canary/MMD 采样──▶│ 证据日志 = 哈希链 + 告警  │ ← 真正的资产
                                              └──────────────────────┘
```

- **数据面**:只做转发 + 流式透传 + tee 副本。开销 ≈ 一跳网络 + 亚毫秒。**用户答案先到手,审计在背后才发生。** 铁律:**fail-open**——审计组件故障时无脑透传、宁丢一条 log 也绝不阻断用户。
- **分析面**:吃 tee 副本,爱算多久算多久。token 重算、指纹、缓存检测、异常打分。做成流式以支撑分钟级告警(如凌晨偷换模型)。
- **探针面**:真实流量缺的信号(LLMmap canary、MMD 采样)靠定时注入合成请求补。
- **证据日志(核心资产)**:append-only + **哈希链**(每条含上条哈希)+ 定期把 Merkle root 对外锚定时间戳。这样中转站改不了、连被攻破的审计器都回改不了历史 → 才能当对账/维权的证据。分层留存:元数据永久;全文仅窗口期或仅对**被标记异常**的请求全留(隐私 + 体积)。

## 5. 与 new-api 结合(最简集成)

核对了 [QuantumNous/new-api](https://github.com/QuantumNous/new-api)(Go 网关,渠道 channel = 上游配置,做 OpenAI/Claude/Gemini 协议转换):

- **关键事实①**:new-api 日志**只存用量,不存完整 body** → "只读它的库做审计"走不通,必须在链路上抓全文。
- **关键事实②**:它**内置"模型映射/重定向"**——降级作弊在 new-api 里本来就是个一行配置的合法功能 → **响应 `model` 字段不可信,真实模型只能靠指纹反推。**
- **关键事实③**:它自带"渠道测试"被社区公认没用(只测 200 通不通,不测真不真)——**这正是我们的楔子。**

**最简集成 = 改一行 `base_url`**:把渠道 base_url 从 `https://upstream` 改成 `http://我们的审计器:8080`(转发到真上游)。**对 new-api 零改动、零 fork**,同样适用 one-api / LiteLLM / 裸客户端。

三种结合深度(优先级):
1. **核心**:独立透传审计代理(改 base_url 挂入)——两个场景通吃、抗规避。**MVP 先做。**
2. **对账**:sidecar 读 new-api DB 的用量/计费,与我方独立重算交叉核对。
3. **上游**:给 new-api 提 PR,把没用的"渠道测试"换成真·身份探测——借它的装机量分发。后期、高杠杆。

## 6. 参考指纹库(护城河,解决硬伤②)

依赖基准的方法(MMD、LLMmap)需要"正品指纹"。基准**不必来自你自己的全量可信上游**(否则直接用官方就行),而来自三个低门槛来源:

1. **偶尔自校准** — 很多买家有一点官方直连(应急卡/免费额度),指纹采样很便宜,偶尔校一次够。
2. **社区签名指纹库(关键)** — 由有官方直连的贡献者共建、多方交叉签名的公开正品指纹库,单点投毒被交叉校验挡掉。每个用户拉公共基准即可。**这跟杀毒特征库、Certificate Transparency 日志是同一个公共品逻辑——是项目最硬的护城河。**
3. **跨中转站共识** — 审计够多家,偏离大众分布的就是嫌疑(诚实多数自动成为基准)。

## 7. MVP 分期

- **Phase 0 — 免基准地板(先发,零可信上游即可用,抓住又懒又贪的多数)**
  - Go 透传式 OpenAI 兼容审计代理(fail-open passthrough + tee)
  - tiktoken 本地重算(OpenAI)→ 计费差异检测
  - 缓存重放检测
  - 哈希链证据日志 + 简易报告/看板
  - 物理速度不可能标记 + 延迟/吞吐采集
- **Phase 1 — 主动探针(需基准,但查询预算低)**
  - LLMmap 式 canary 探针注入(探针面),候选集闭集分类
  - 参考指纹来自"偶尔自校准"
- **Phase 2 — 统计分布审计(深度方法 + 护城河)**
  - 对真实流量采样 + 探针跑 MMD model-equality 检验
  - 社区签名参考指纹库
- **Phase 3 — 生态**
  - new-api PR(真·渠道身份测试)
  - 多厂商用量(Claude `count_tokens`、Gemini)

## 8. 诚实的限制(必须摆在明处)

- **归因鸿沟(最重要)** — 阳性分布检验只证明"与参考权重不一致",**不能干净区分"恶意替换" vs "良性量化/微调/水印/推理引擎非确定性"**。源论文刻意避免纯诈骗框架。我们输出"差异 + 证据",由人/社区裁决。
- **用量验证不对称** — 仅 OpenAI 可完全离线核验;Claude/Gemini 无公开分词器,退化为"信转发数字"或"往返厂商拿仅输入的估计"。硬限制。
- **无密码学地面真相** — 没有厂商端到端签名响应。终局(单请求级铁证)= 厂商签名 / TEE 可信执行(NVIDIA 机密计算等,**研究中未验证到 3 票强度,前景有但不能等**)/ zkML。我们应推动该开放标准,但不阻塞 MVP。
- **数字别神化** — MMD 77.4% 是**仿真中位**(微调远难于量化);LLMmap >95% 是**闭集**分类(未见过的替换模型落入更难的开集)。现实对抗下更低。
- **逐项可被针对性规避** — 故必须 defense-in-depth + 被动全量(让它分不清哪条在被审)。
